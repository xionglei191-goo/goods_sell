### 2.8 渠道经营扩展模型（Phase 7 规划）

现有模型已覆盖订单、分单、客户、画像、聊天、优惠券和经销商库存，但要承接“业务员地推 + 经销商私域导流 + AI互动画像 + 大单询价报价”的新定位，还需要新增以下模型：

| 模型/字段 | 作用 | 关键关系 |
|-----------|------|----------|
| Lead | 记录扫码、AI互动、询价、注册但未下单的线索 | 关联 Customer、User(业务员)、Dealer、PromoterCode |
| Inquiry | 记录宴席、团购、门店补货等非标准购买需求 | 可由 Lead 创建，后续转 Quote 或 Order |
| Quote | 记录业务员/公司给出的报价方案、商品组合、有效期 | 关联 Inquiry、Order、User |
| PromoterCode | 业务员/经销商专属二维码和来源追踪 | 关联 User、Dealer、Campaign |
| DealerPolicy | 经销商接单政策、可接品牌、金额边界、价格权限 | 关联 Dealer、Brand |
| ProductPush | 新品推送记录、命中画像/客户分层、触达事件、转化和复盘状态 | 关联 Product、Customer、Campaign |
| ChannelConflict | 跨区下单、异常低价、拒单、投诉台账 | 关联 Order、Dealer、Customer、User |

建模原则：`Lead`、`Inquiry` 和 `Order` 必须分开。大单、团购和宴席采购通常需要先确认库存、报价、配送、发票和账期，不能直接套普通购物车支付流程。

## 4. Phase 10：权限与经销商申请数据约定

本阶段不新增数据库迁移，复用现有账号、客户、线索和经销商模型完成角色化体验。

| 场景 | 数据落点 | 状态约定 |
|------|----------|----------|
| 消费者注册 | `Customer(type=CONSUMER,isVerified=true)` | 注册后可登录商城 |
| 经销商申请 | `Customer(type=DEALER,isVerified=false)` + `Lead(scene=DEALER_JOIN,source=SHOP,status=NEW)` | 待管理员审核，不能登录 |
| 经销商审核通过 | 创建 `Dealer`，更新 `Customer.isVerified=true`，更新 `Lead.status=CONVERTED` | 可登录 `/dealer/*` |
| 经销商审核驳回 | 保留 `Customer` 和 `Lead`，更新 `Lead.status=LOST`，在 `Lead.metadata` 记录驳回原因 | 不可登录 |

`Lead.metadata` 用于保存申请扩展信息：`shopName`、`zone`、`address`、`businessLicense`、`contactName`、`approval/rejection` 操作记录。`Dealer` 的 `latitude`、`longitude`、`serviceRadius`、`zone` 仍为正式门店档案必填字段，由管理员审核时补齐。

权限矩阵采用代码配置，不新增数据库表。服务端根据 `User.role` 或 `Customer.type` 派生应用角色：`ADMIN`、`SALESPERSON`、`WAREHOUSE`、`FINANCE`、`DEALER`、`CONSUMER`。
