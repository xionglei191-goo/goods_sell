# 华启酒饮数字渠道平台 — AI Tools 与代办助手说明

## 1. 能力定位

Phase 11 将原有“AI 选品互动”升级为“AI 代办入口 + 权限化 tools 执行层”。系统不再只让 AI 回答推荐问题，而是让用户通过自然语言发起查询、草稿、确认和执行操作。

典型目标：

- 客户说“我要下单 1 箱青岛经典啤酒”，AI 自动匹配商品、读取默认地址、生成下单确认卡，客户确认后提交订单或询价。
- 老板说“这个月李明的业绩怎么样”，AI 按管理员权限调用销售员业绩 tool，返回销售额、订单数、客户数、回款和排名摘要。
- 仓管说“给可口可乐入库 20 箱”，AI 生成入库确认卡，确认后复用库存 action 执行并写入日志。
- 管理员说“检查现在上线还差什么配置”，AI 调用上线检查 tool，返回数据库、认证、AI、地图、微信、票税和酒类资质配置状态。

设计原则：

- AI 不绕过业务逻辑直接写数据库。
- 所有 tool handler 优先复用现有 `features/*/actions.ts`、`queries.ts`、权限守卫和 `logAction()`。
- 所有写操作必须确认；高风险操作必须二次确认。
- 权限以当前登录身份为准，AI 不展示、不执行越权数据。

## 2. 用户入口

| 入口 | 组件/路由 | 覆盖对象 | 说明 |
|------|-----------|----------|------|
| AI 悬浮气泡 | `src/features/ai/AiFloatingBubble.tsx` | 商城、后台、经销商端 | 对话、结果卡、确认卡、高风险确认输入 |
| 商城端接入 | `src/features/shop/ShopShell.tsx` | CONSUMER | 商品查询、客户信息读取、下单草稿、订单查询、应收查询 |
| 后台端接入 | `src/components/layout/DashboardShell.tsx` | ADMIN、SALESPERSON、WAREHOUSE、FINANCE | 经营查询、客户/订单/商品/库存/财务/营销/系统操作 |
| 经销商端接入 | `src/features/dealer/DealerShell.tsx` | DEALER | 待接单、接单/拒单、库存上报、结算查询 |

气泡 UI 使用“对话 + 操作卡片”模式：

- 固定词条按角色和当前数据动态显示：消费者、经销商、管理员、销售员、仓管、财务各自只看到本角色常用、有权限且已通过预检的快捷问法。
- 查询类 tool 返回摘要卡和关键字段。
- 写操作返回确认卡，用户确认后调用确认 API。
- 高风险操作要求输入指定确认文字，避免误触。
- 文案、按钮和字号按适老化使用场景设计，减少菜单层级和重复输入。

## 3. API 接口

| 接口 | 方法 | 用途 |
|------|------|------|
| `/api/ai/chat` | POST | 保留原商城 AI 选品/场景互动接口，避免破坏既有页面 |
| `/api/ai/assistant` | POST | 新 AI 气泡对话接口，支持 `message`、`quickPromptId?`、`pathname?`，返回 SSE 事件 |
| `/api/ai/assistant/confirm` | POST | 执行待确认 tool，校验确认 token 和高风险确认文字 |
| `/api/ai/quick-prompts` | GET | 按当前登录角色返回已验证固定词条；未登录返回空词条 |

`/api/ai/assistant` SSE 事件：

| 事件 | 含义 |
|------|------|
| `delta` | AI 文本回复片段 |
| `status` | 固定词条验证、工具筛选、参数规划、计划校验、计划修复、工具调用、确认卡生成、结果整理等阶段提示 |
| `card` | 查询结果卡或待确认操作卡 |
| `done` | 本轮对话完成，返回 `planSource` 便于排查模型/模板/启发式来源 |
| `error` | 权限、登录、参数或执行失败 |

未登录调用 assistant API 时返回 SSE `error` 事件，状态为 `401`，前端展示“请先登录后再使用 AI 助手”。

## 4. Tool 定义规范

Tool 统一定义在 `src/features/ai/tools/registry.ts`，类型在 `src/features/ai/tools/types.ts`。

```typescript
type AiToolDefinition = {
  name: string
  title: string
  description: string
  category?: "NAVIGATE" | "READ" | "DRAFT" | "WRITE" | "HIGH_RISK"
  href?: string
  capabilityIds?: string[]
  capabilities?: string[]
  examples?: string[]
  argumentHints?: string
  requiredSlots?: string[]
  inputSchema: z.ZodTypeAny
  riskLevel: "READ" | "DRAFT" | "WRITE" | "HIGH_RISK"
  access?: {
    roles?: AppRole[]
    permission?: DashboardPermission
  }
  resolvePermission?: (input, context) => DashboardPermission | null
  handler: (input, context) => Promise<AiToolResult>
  buildConfirmation?: (input, context) => Promise<ConfirmationCard>
}
```

执行链路：

1. `context.ts` 读取当前登录用户、角色、账号类型和数据范围。
2. `intent-templates.ts` 基于当前角色、真实商品/销售员/订单等数据生成候选固定词条，并通过 `preflightAiTool()` 验证后才展示。
3. 固定词条点击时提交 `quickPromptId`，服务端重新验证后直接生成 tool plan。
4. 普通自然语言进入 Planner v2：`model-planner.ts` 先按当前角色可用 tools 的 `capabilities/examples/argumentHints` 进行 top-k 排序，再把候选交给真实 AI provider 输出结构化 JSON plan。
5. provider plan 必须经过工具名、参数 schema、角色权限、动态权限、核心意图和写操作确认卡预检；失败时带原因进行一次模型修复。
6. `planner.ts` 作为本地降级与纠偏层，provider 不可用、修复失败或核心意图错配时再接管；低置信且本地无高置信命中时返回澄清卡，而不是展示工具清单。
7. `executor.ts` 执行参数校验、权限校验、动态权限判断、确认卡生成、handler 调用和错误归一。
8. `audit.ts` 记录 assistant 过程，并对密码、token、secret 等敏感字段脱敏；`prompt-usage.ts` 记录非固定高频候选。
9. Confirm API 校验 HMAC 签名 token、用户、角色、tool、参数、风险等级和 10 分钟有效期。

意图识别边界：

- 固定词条用于高频、低歧义问法，但必须引用当前数据库中真实存在的对象；例如商品词条使用当前上架商品，不再硬编码不存在的“剑兰春”。
- 本地规则只处理高置信场景，禁止用单个“买”等宽泛关键词直接触发写操作。
- “我要下单/帮客户开单/要 N 箱”属于下单或开单意图；“某客户买了什么/买过什么/购买记录”属于客户购买历史查询。
- 含糊表达和普通自然语言优先交给真实 AI provider；provider 返回结果仍需经过 tool 名称、参数 schema、权限和核心意图校验，失败后回退本地规则。
- “库存有多少/库存最多/低库存/快没货”属于 `product_operations_summary`；经销商“上报/报库存/门店库存为 N”才属于 `dealer_report_stock`。
- “现在一共有多少客户/哪个客户消费最高/客户总数”属于 `customer_analytics_summary`；经营总览中的“客户数”只表示本期新增客户，不能替代客户总量分析。
- “谁欠款最多/应收账龄/回款趋势”属于 `finance_summary`；“给订单登记收款 N 元”才属于 `finance_register_payment`。
- “李明最近转化怎么样/销售员成交如何”属于 `salesperson_performance`，由 provider 从排序后的候选工具中规划参数。
- 高频候选只记录在 `AiContentCache` 中，人工审核后再进入代码内置固定词条，不自动上线。

## 5. 风险分级

| 等级 | 执行规则 | 示例 |
|------|----------|------|
| READ | 可直接执行 | 查商品、查订单、查销售员业绩、查应收 |
| DRAFT | 只生成草稿，不落库或不改变关键业务状态 | 新增商品草稿、后台开单草稿、优惠券草稿 |
| WRITE | 生成确认卡，用户确认后执行 | 下单、调价、入库、出库、登记收款、发券 |
| HIGH_RISK | 确认卡 + 指定文字二次确认 + 审计日志 | 禁用账号、重置密码、系统参数修改、财务确认 |

所有 `WRITE` 和 `HIGH_RISK` tool 都会写入审计日志。AI 回复不暴露越权数据；越权时返回“无权限查看/操作”。

## 7. 权限边界

| 角色 | AI tools 范围 |
|------|---------------|
| ADMIN | 可使用全部后台查询、管理、营销、系统和高风险 tools |
| SALESPERSON | 只暴露客户、订单、线索、询价、报价、销售业绩等权限内能力；数据默认限制为名下或分派范围 |
| WAREHOUSE | 只暴露库存、采购、仓储、配送、订单履约相关能力 |
| FINANCE | 只暴露财务、收款、票据、对账和订单/客户只读能力 |
| DEALER | 只暴露自己的待接单、接单/拒单、库存上报、结算和线索能力 |
| CONSUMER | 只暴露自己的商品咨询、下单、订单、地址和账务信息 |

权限校验分三层：

- 前端气泡只展示当前登录用户可用固定词条和入口。
- assistant runtime 只向 planner 暴露当前角色可用 tools。
- executor 在服务端再次校验角色、`DashboardPermission` 和动态权限。
- `customer_purchase_history` 仅对 ADMIN、SALESPERSON、FINANCE 开放；销售员查询时仍限制为名下客户，仓管、经销商和消费者不能通过 AI 查看他人购买历史。

## 10. 全站 Agent Capability 目录

本轮在 tools 层前新增 `AgentCapabilityRegistry`，文件位于 `src/features/ai/tools/capabilities.ts`。它不是替代业务页面，而是让 AI 先理解“用户要找哪个功能”，再决定调用可执行 tool、返回页面入口，或给出功能帮助。

能力分层：

| 类型 | 行为 | 示例 |
|------|------|------|
| NAVIGATE | 返回当前角色可进入的页面入口，不改数据 | “供应商管理在哪”“微信菜单在哪配置” |
| READ | 查询摘要，仍走现有 tool handler | “谁欠款最多”“库存最多的是哪个商品” |
| DRAFT | 生成草稿，不落库 | “做一张优惠券草稿” |
| WRITE | 生成确认卡，确认后执行 | “给商品入库 10 件” |
| HIGH_RISK | 二次确认并审计 | “禁用员工账号”“修改系统参数” |

Planner v2 的新顺序：

1. 读取当前登录用户和角色。
2. 过滤当前角色可用 tools，并同时按 `AgentCapabilityRegistry` 匹配全站页面能力。
3. 把 top-k tools 和 top capability 候选一起交给真实 AI provider。
4. provider 若判断是“在哪/打开/怎么配置/页面帮助”，优先规划 `navigate_to_feature` 或 `feature_help`，并返回 `capabilityId`。
5. provider 不可用或低置信时，本地 `planAgentCapabilityNavigation()` 作为兜底，确保功能入口类问题仍能命中。
6. provider 不可用时，`planRankedReadToolFallback()` 会对高置信 READ tools 做本地语义兜底，覆盖采购供应商、库存流水、微信生态、购物车等查询场景。
7. 数据查询和写操作仍优先走已有业务 tools，导航工具不得抢占“欠款排行、客户统计、库存总览”等可执行查询。

新增通用 tools：

| tool | 风险 | 入参 | 返回 |
|------|------|------|------|
| `navigate_to_feature` | READ/NAVIGATE | `{ query?, capabilityId? }` | 功能标题、说明、入口 `href`、模块、权限与关联 tool |
| `feature_help` | READ/NAVIGATE | `{ query?, capabilityId? }` | 功能说明、当前角色是否可进入、示例问法 |

新增验收：

- `npm run test:agent-capabilities` 扫描全部 `page.tsx`，要求每个页面至少有 capability 覆盖。
- “供应商管理在哪”“怎么查看库存流水”“帮我打开经销商政策”“查购物车”“微信菜单在哪配置”必须命中 `navigate_to_feature`。
- “谁欠款最多”“现在一共有多少客户，哪个消费最高”“现在库存有多少商品，哪个库存最多”必须继续命中原业务查询 tool。

## 11. 扩展新 Tool 或新功能入口的步骤

1. 先确认现有 `features/*/actions.ts` 或 `queries.ts` 是否已有业务能力，优先复用。
2. 在 `registry.ts` 新增 tool 定义，补齐 `name`、`title`、`description`、`inputSchema`、`riskLevel`、`access` 和 `handler`。
3. 补 `capabilities`、`examples`、`argumentHints`，让 Planner v2 能通过语义排序把该工具放进 top-k 候选。
4. 写操作补 `buildConfirmation()`；高风险操作补 `confirmTextRequired`。
5. 若权限与参数有关，使用 `resolvePermission()` 做动态权限判断。
6. 只有高置信、强业务边界的场景才在 `planner.ts` 增加本地纠偏规则；普通泛化问法优先靠 tool 语义元数据和 provider 规划。
7. 如果新增页面、菜单或暂未 tool 化的业务入口，同步在 `capabilities.ts` 增加 `AgentCapability`，声明 `href`、`paths`、角色/权限、关键词和示例问法。
8. 更新 `scripts/ai-tools-smoke.ts`、`scripts/agent-capabilities-smoke.ts`，必要时更新 `scripts/ai-provider-smoke.ts` 场景覆盖。
9. 跑 `npm run test:agent-capabilities`、`npm run test:ai-tools`、`npm run test:ai-runtime`、`npm run lint`、`npm run build`。
