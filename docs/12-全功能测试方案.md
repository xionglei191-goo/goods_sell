## 2. 测试目标

本方案用于验证华启酒饮数字渠道平台在正式运营前是否具备可用、可控、可追溯的上线条件。

测试目标包括：

1. 验证客户从进入商城、提交需求、下单、支付、履约到售后的主流程可闭环。
2. 验证公司内部商品、库存、采购、订单、配送、财务和对账流程可正常使用。
3. 验证经销商分单、接单、拒单、库存上报和结算流程符合渠道保护策略。
4. 验证线索、询价、报价、推广码、新品推送和 AI 互动能支持运营获客。
5. 验证权限、日志、合规提示、异常处理和数据口径满足试运营要求。
6. 形成上线准入结论，明确必须修复项、可延期项和试运营风险。

## 3. 测试原则

- 先测主链路，再测边界和异常。
- 先测 P0 运营阻断问题，再测 P1/P2 体验与运营问题。
- 所有涉及金额、库存、分单、收款和客户资料的流程必须有可追溯记录。
- 测试数据应与真实业务接近，但不得混入真实客户隐私数据。
- 测试通过不代表直接全面运营，建议先进入封闭试运营。

## 8. 缺陷等级

| 等级 | 定义 | 处理要求 |
|------|------|----------|
| S1 阻断 | 系统无法登录、下单、收款、库存错误、数据丢失、越权访问、支付错误 | 上线前必须修复 |
| S2 严重 | 主流程可走但结果错误、分单错误、财务口径错误、订单状态异常 | 上线前必须修复或有明确人工兜底 |
| S3 一般 | 单个功能异常、不影响核心运营、有替代操作 | 可进入试运营，但需排期修复 |
| S4 建议 | 文案、样式、交互优化、低频体验问题 | 不阻断上线 |

## 9. 上线准入标准

满足以下条件才建议进入封闭试运营：

1. 所有 P0 测试用例通过率达到 100%。
2. 不存在未关闭的 S1 缺陷。
3. S2 缺陷已修复，或业务负责人确认有人工兜底方案。
4. 商品、库存、经销商、价格和客户测试数据通过抽查。
5. 小单经销商承接、大单公司报价、拒单转派三条主链路通过。
6. 收款、应收、对账和订单金额口径一致。
7. 操作日志可追溯关键操作。
8. 客户侧合规提示和个人信息授权可见。
9. 已完成一次模拟运营演练。
10. 已明确试运营范围、参与人员、应急联系人和回滚方案。

## 15. 模拟运营演练

上线前建议组织一次 2-4 小时模拟运营演练。

### 15.1 参与人员

- 管理员 1 人。
- 销售员 2 人。
- 仓管 1 人。
- 财务 1 人。
- 经销商 2-3 家。
- 测试客户 5-10 人。
- 开发或运维 1 人值守。

### 15.2 演练剧本

1. 普通客户下小单，经销商 A 接单并完成配送。
2. 普通客户下小单，经销商 B 拒单，系统转派或转公司处理。
3. 宴席客户提交大单询价，销售员创建报价并转订单。
4. 企业客户提交团购询价，要求开票和分批配送。
5. 门店客户提交补货需求，业务员跟进并转订单。
6. 仓管执行入库、出库和盘点。
7. 财务登记一笔现金收款、一笔转账收款、一笔赊账核销。
8. 运营创建优惠券并让客户使用。
9. 运营执行一次新品推送并标记转化。
10. 管理员处理一条渠道冲突并关闭。

### 15.3 演练通过标准

- 每个剧本均能完成闭环。
- 金额、库存、订单状态和客户归属无明显错误。
- 操作人员能理解自己的入口和待办。
- 所有异常有人工兜底方案。
- 演练问题已记录负责人和修复计划。

## 18. 试运营建议

全功能测试通过后，建议不要立即全面开放，而是先进入封闭试运营：

| 阶段 | 范围 | 时间 | 目标 |
|------|------|------|------|
| 内部试用 | 公司内部人员 | 1-2 天 | 熟悉系统、发现明显问题 |
| 小范围试运营 | 2-3 家经销商、10-30 个客户 | 3-7 天 | 验证真实下单、分单、收款和配送 |
| 扩大试运营 | 10 家以内经销商、100 个以内客户 | 1-2 周 | 验证运营承接能力和异常处理 |
| 正式运营 | 全部目标客户和经销商 | 根据前期结果决定 | 稳定运营和数据复盘 |

试运营期间应每日复盘：
- 新增线索、询价、订单数量。
- 小单经销商承接率。
- 拒单和渠道冲突。
- 库存准确率。
- 收款和应收差异。
- 客户反馈。
- 系统错误和慢接口。

## 19. 上线风险清单

| 风险 | 影响 | 建议 |
|------|------|------|
| 商品价格错误 | 直接造成损失或渠道投诉 | 上线前双人复核价格 |
| 库存不准 | 超卖、缺货、配送失败 | 上线前盘点核心商品 |
| 分单规则不准 | 经销商不满或客户体验差 | 先小范围验证经销商半径和政策 |
| 财务口径不一致 | 对账困难 | 财务参与测试并确认口径 |
| 微信支付配置错误 | 无法付款或回调失败 | 先做小额真实支付验证 |
| 客户隐私授权不足 | 合规风险 | 明示收集用途和授权 |
| 酒类营销文案违规 | 监管和品牌风险 | 上线前人工审核页面和 AI 话术 |
| 运营人员不熟悉 | 真实订单处理延误 | 做角色培训和演练 |
| 缺少应急方案 | 问题扩大化 | 准备人工接单、手工对账和回滚方案 |

## 21. Phase 10 RBAC 专项测试

| 用例 | 预期结果 |
|------|----------|
| 未登录访问 `/dashboard/orders` | 跳转 `/login?callbackUrl=/dashboard/orders` |
| 消费者访问 `/dashboard` | 跳转 `/forbidden` |
| 经销商访问 `/dashboard` 或 `/shop/cart` | 非本角色入口跳转 `/forbidden` |
| 销售员登录后台 | 不显示库存、采购、仓储、财务、票据、设置、日志菜单 |
| 仓管登录后台 | 只显示产品只读、库存、采购、订单履约、仓储、配送等菜单 |
| 财务登录后台 | 显示订单/客户只读、财务、票据、销售报表，不显示仓储和渠道经营写操作入口 |
| 消费者注册 | 注册成功后可登录并进入 `/shop` |
| 经销商申请 | 创建 `DEALER_JOIN` 线索，账号不可登录 |
| 管理员审核通过经销商 | 创建 `Dealer` 档案，申请线索变为 `CONVERTED`，经销商可登录 `/dealer/incoming` |
| 管理员驳回经销商 | 申请线索变为 `LOST`，账号仍不可登录 |
| 销售员访问客户/线索/订单 | 只能看到自己名下或分派数据 |
| 错误角色调用库存/财务/设置 action | 返回 `UNAUTHORIZED` 或无权限错误 |

自动化验证需增加 RBAC smoke 脚本，并与 `npx tsc --noEmit`、`npm run lint`、`npm run build`、`npm audit --audit-level=moderate` 一起作为上线前检查。
